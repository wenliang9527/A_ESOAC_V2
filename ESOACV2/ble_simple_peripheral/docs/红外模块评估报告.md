# 红外控制模块代码评估报告

对 **ESOACV2/ble_simple_peripheral** 中红外学习/发送实现进行代码级评估，覆盖功能闭环、协议实质、三通道控制与稳定性风险；结论以仓库现状为准。本文档由计划整理落地，不修改源代码。

**评估范围**：`usercode/frIRConversion.c`、`usercode/frIRConversion.h`、`usercode/IRLearntr.c`、`usercode/protocol.c`、`usercode/protocol.h`、`usercode/ESAIRble_service.c`、`usercode/mqtt_handler.c`、`usercode/frusart.c`、`usercode/app_task.c`。

---

## 1. 功能完整性

### 1.1 已实现能力

| 环节 | 实现要点 | 完成度 |
|------|----------|--------|
| 红外学习 | `IR_start_learn()` 配置 PC3/EXTI_3 + Timer0 1ms；`exti_isr_ram` 采边沿间隔；`timer0_isr_ram` 内 `IR_WAIT_STOP` / `IR_LEARN_GET_DATA` 状态机；`calculate_carrier_frequency`、`parse_pulse_data`、`ir_data_check` 后 `save_learned_data` 写入 `ESairkey.airbutton[AIPstudyKey]` | 学习链路完整 |
| 学习触发与停止 | `ESOAAIR_IRskeystudy` 启动 10s 定时器调用 `IR_stop_learn`；协议 `CMD_IR_LEARN_START` / `CMD_IR_LEARN_STOP` 对接上述 API | 完整 |
| 发送已学习波形 | `ESOAAIR_IRsend` 从 `airbutton[key]` 取时序与载波，`IR_start_send` 用 Timer0 微秒节拍切换 PWM5 调制 | 完整 |
| 业务侧“按键”控制 | `protocol.c` 中 `CMD_SET_POWER/MODE/TEMP/WIND` 通过 `air_ir_send_key_and_wait` 映射到键号 0~4 并调用 `ESOAAIR_IRsend` | 完整 |
| 学习数据落盘 | `ESOAAIR_Savekey()` 将整块 `ESairkey` 写入 SPI Flash（`W25QADDR2`，见 `usercode/frspi.h`）；协议 `CMD_IR_SAVE_KEYS` 调用 | 写入路径存在 |

### 1.2 闭环缺口与功能风险

- **上电恢复**：`ESOAAIR_readkey()` 仅在 `usercode/IRLearntr.c` 实现，**工程内未见任何调用点**；若不补充启动时加载，则“保存—断电—再发”链路在运行时语义上**不完整**。
- **发送完成收尾**：`IR_init()` 负责创建 `IR_task_func`，发送结束时 `timer0_isr_ram` 通过 `os_msg_post(user_ir_task_id, …)` 通知任务释放 PC5、清 `IR_Busy`。**当前源码树中无 `.c` 调用 `IR_init()`**，存在 `user_ir_task_id == TASK_ID_NONE` 时事件投递无效、GPIO/PWM 状态与 busy 标志无法按设计清理的风险（与链接裁切行为一致时需实测确认）。
- **声明未实现**：`process_ir_data()` 为空；`ir_data_to_int()` 仅在头文件声明，主工程 `frIRConversion.c` 无实现——不影响当前“原样回放”路径，但表明无二次解码能力。
- **IR_decode 路径**：固定引导码/高低电平展开存在，但**主业务未走该路径**；空调控制依赖学习回放。

**小结**：学习 + 内存中发送 + Flash 写入在代码层面齐全；**持久化读出与 IR 任务初始化**在集成层面存在明显缺口，完整产品级工作流应视为**部分完成**。

## 2. 协议兼容性（NEC / RC5 / RC6 及主流空调）

### 2.1 代码实际模型

- **主路径**：**载波包络 + 间隙时间的录制与回放**，载波频率粗分为 36k / 38k / 56k（`calculate_carrier_frequency`），并非按协议名解析帧结构。
- **校验** `ir_data_check`：基于间隔落在约 12ms~100ms 的“停止位”数量、过短脉冲、长间隔与重复帧长度等**启发式规则**，并含对特定重复结构的注释兼容（如 Gemini-C10）。这是**经验性**判定，不是标准 NEC/RC5/RC6 解码器。
- **IR_decode**：时间常量（如 9ms/4.5ms 引导、560µs 等）接近常见 **NEC 类**波形描述，但**业务命令未使用该函数生成空调码**。

### 2.2 与“主流品牌协议”的关系

- **未实现**独立的 NEC 地址/命令反相校验、RC5 双相位、RC6 长头与模式字段等**标准栈**。
- 对**各品牌空调**：多数使用长帧、多段、反码或动态协议；本实现依赖**学习波形能否通过 `parse_pulse_data` + `ir_data_check`** 以及定时精度。部分机型可能学习失败、回放偏移或重复键行为异常。
- **结论**：不能表述为“支持 NEC/RC5/RC6 等编码格式的完整兼容”；更准确是 **“通用红外学习回放 + 类 NEC 时序的可选编码函数（未接入主业务）”**。对市面空调的覆盖取决于**实测**，而非代码中的协议枚举。

## 3. 控制接口支持（BLE / MQTT / UART）

### 3.1 统一协议入口

- 三种来源均解析**同一二进制帧**（`protocol_parse_frame`），并进入 `protocol_process_frame(frame, source)`：
  - **BLE**：`app_task.c` 中 `APP_EVT_BLE_DATA_RECEIVED` → `source = 0`。
  - **MQTT**：`mqtt_handler.c` 注释明确与 UART/BLE 相同帧 → `source = 1`。
  - **UART**：`frusart.c` 在 `frUSART1 == 1` 时定时解析 → `source = 2`。

### 3.2 红外相关命令可达性

- `CMD_IR_LEARN_START`、`CMD_IR_LEARN_STOP`、`CMD_IR_SEND`、`CMD_IR_READ_DATA`、`CMD_IR_SAVE_KEYS` 均在 `protocol_process_frame` 的 `switch` 中处理，**不区分 source**，故理论上 **BLE / MQTT / UART 均可下发**（前提是各通道确实把完整帧送达解析器）。

### 3.3 响应回传差异

- `protocol_send_buffer`：BLE 走 GATT Notify；MQTT 走 `ML307A_MQTTPublish`（需模组已 MQTT OK）；UART 为 **`uart_write(UART1, …)` 回传**（注释为调试/桥接）。**接口对称性**：命令侧统一，应答侧依赖链路能力（如 MQTT 未连接则打印告警并跳过发布）。

### 3.4 MQTT 发布辅助代码的完整性说明

- `mqtt_handler.c` 中 `mqtt_handler_publish` 仍含“需将二进制转 HEX / 依模组 AT 格式调整”的占位说明；**与红外无直接冲突**，但表明 MQTT 数据路径需与 ML307A 实际 AT 手册对齐后做验收。

**小结**：**逻辑上**已具备三通道控制空调与红外学习/发送的能力；工程上需验证 MQTT/UART 物理链路与帧边界（UART 10ms 定时聚帧等）是否满足产品场景。

## 4. 系统稳定性与多任务

### 4.1 积极设计

- 发送与学习通过 `ir_mode`、`IR_learn_state` bit、`IR_Busy` 等做**互斥**（学习中拒绝 `IR_start_send`，发送忙拒绝重入）。
- 空调控制命令在发码前使用 `air_ir_send_key_and_wait` **等待发送结束**，降低命令重入导致时序错乱的概率。
- Timer0 ISR 中短暂 `NVIC_DisableIRQ(BLE_IRQn)` 再改 PWM 寄存器，意图避免与协议栈并发写硬件。

### 4.2 风险点

- **`air_ir_send_key_and_wait` 忙等待**：在应用任务上下文中轮询 `IR_PWM_TIM.IR_Busy` + `co_delay_100us`，长时间会占用任务调度份额，高负载下可能影响同任务内其它事件响应（需结合 OS 配置评估）。
- **全局共享状态**：`ESairkey`、`ir_learn_data`、学习过程与协议解析**无显式锁**；正常单连接、顺序命令下可接受，并发写或多连接若未来扩展需防护。
- **动态内存**：每次 `IR_start_learn` 使用 `os_zalloc(sizeof(TYPEDEFIRLEARN))`，`IR_stop_learn` 中 `os_free`；长期反复学习需关注**堆碎片**（取决于 RTOS 堆实现）。
- **Timer0 单一资源**：发送与学习复用 Timer0 + 部分 NVIC 优先级调整；若其它模块误用 Timer0 会产生冲突（当前以 IR 模块独占为前提）。
- **发送完成路径依赖 `IR_init`**：如前所述，缺失初始化时存在**资源泄漏式状态**（busy、引脚复用）风险。

### 4.3 长时间运行

- 无专门看门狗或与红外绑定的老化逻辑；稳定性更多取决于 BLE 栈、4G 模组与上述并发/ISR 设计。红外子系统本身**无周期性自愈**（除学习失败重回 `IR_WAIT_STOP`）。

---

## 5. 潜在问题汇总（优先级建议）

1. **高**：启动流程增加 `IR_init()`；上电调用 `ESOAAIR_readkey()`（或在 Flash 初始化后统一加载 `ESairkey`）。
2. **高**：对 `os_msg_post(TASK_ID_NONE, …)` 做代码审查或断言，避免静默失败。
3. **中**：将 `air_ir_send_key_and_wait` 改为事件驱动或带超时的非阻塞状态机，改善多任务友好性。
4. **中**：产品级声明“支持空调品牌”前，按目标品牌做**学习/回放**矩阵测试，并记录 `ir_data_check` 失败样例以迭代启发式规则。
5. **低**：补齐或删除未实现 API（`IR_test_demo0`、`ir_data_to_int`、`process_ir_data`），减少集成误导。

---

## 6. 总表

| 维度 | 结论（简要） |
|------|----------------|
| 功能完整性 | 学习+内存发送+Flash 写齐全；**缺上电加载与 IR 任务初始化**时闭环不完整 |
| 协议覆盖 | **非**标准 NEC/RC5/RC6 栈；以**学习回放**为主，品牌兼容靠实测 |
| 接口兼容 | BLE/MQTT/UART 共用 `protocol_process_frame`，红外命令**三通道可达** |
| 稳定性 | 基础互斥具备；忙等待、全局无锁、`IR_init` 缺失为**主要可靠性隐患** |
