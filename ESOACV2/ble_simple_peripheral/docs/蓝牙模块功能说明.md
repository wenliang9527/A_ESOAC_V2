# 蓝牙（BLE）模块功能说明

本文档描述 ESOACV2 工程中蓝牙低功耗（BLE）通讯相关软件实现，覆盖芯片驱动层（`fr8010h_stack` 协议栈）、GAP/GATT 服务层（`ble_simple_peripheral`、`ESAIRble_service`）与应用协议层（`protocol`），并与当前代码保持一致，供开发与维护查阅。

---

## 1. 功能概述

### 1.1 协议栈层（BLE Stack）

基于 FR8010H 平台 BLE 5.0 硬件与官方协议栈库（`libfr8010h_stack.a`），提供：

- **工作模式**：从机（Peripheral）模式，支持单连接。
- **GAP 功能**：广播（Advertising）、连接参数管理、配对与绑定、安全加密。
- **GATT 功能**：服务（Service）与特征值（Characteristic）定义、读写操作、通知（Notification）。
- **物理层**：支持 LE 1M PHY，默认 0dBm 发射功率。
- **绑定管理**：支持 Flash 持久化存储配对密钥（LTK/IRK）与对端服务信息。

### 1.2 GAP 层（`ble_simple_peripheral`）

本工程 GAP 层封装提供：

- **广播管理**：可配置广播间隔、广播数据、扫描响应数据；支持动态更新设备名称并重启广播。
- **连接管理**：处理连接建立、断开、参数更新事件；支持连接后自动请求更优连接参数。
- **安全策略**：采用 Just Works 配对（无 MITM、无输入输出能力），启用绑定与密钥存储。
- **设备标识**：使用芯片 eFuse 唯一 ID 生成公共地址（Public Address），设备名称格式为 `ESOAC` + 12 位十六进制地址。

### 1.3 GATT 服务层（`ESAIRble_service`）

本工程定义自定义 ESAIR 服务，用于空调节能终端的蓝牙透传通讯：

- **服务 UUID**：`02F00000-0000-0000-0000-000000FE0010`
- **特征值**：
  - **Version Info**（`0xFF13`）：只读，协议版本号（当前 0x0001）。
  - **Notify CCC**（`0xFF12`）：客户端特征值配置描述符（CCC），用于使能/禁用通知。
  - **TX Data**（`0xFF10`）：通知特征值，设备→手机数据通道。
  - **RX Data**（`0xFF11`）：写入特征值，手机→设备数据通道。
- **数据长度**：单包最大 128 字节（`ESAIR_MAX_DATA_SIZE`）。

### 1.4 OTA 服务层（`ota_service`）

提供空中升级（Over-The-Air）支持：

- **服务 UUID**：与 ESAIR 服务共用 `0xFE00` 基址，独立特征值定义。
- **升级流程**：通过 RX 特征值接收升级数据，TX 特征值返回升级状态与进度。
- **安全机制**：升级数据校验和验证，支持断点续传。

## 2. 核心工作流程

### 2.1 系统初始化流程

BLE 初始化入口位于 `user_entry_after_ble_init`，在系统启动后、主循环前执行：

```c
void user_entry_after_ble_init(void)
{
    // ... 其他初始化 ...
    
    // 1. GAP 层初始化：设备名称、广播参数、安全策略
    simple_peripheral_init();
    
    // 2. 添加 OTA GATT 服务
    ota_gatt_add_service();
    
    // 3. 添加 ESAIR 主 GATT 服务
    ESAIR_gatt_add_service();
    
    // ... MQTT 等后续初始化 ...
}
```

主要步骤（逻辑顺序）：

1. **设备名称初始化**：优先从 Flash 加载已保存名称；若不存在则基于芯片公共地址生成默认名称 `ESOACXXXXXXXXXXXX`。
2. **GAP 参数配置**：设置设备外观（Generic Remote Control）、安全参数（无 MITM、等待配对请求）、绑定管理器（最多 8 个绑定设备）。
3. **注册事件回调**：`gap_set_cb_func(app_gap_evt_cb)` 注册 GAP 事件处理函数。
4. **启动广播**：所有 GATT 服务添加完成后，触发 `GAP_EVT_ALL_SVC_ADDED` 事件，自动开始广播。

### 2.2 广播流程：`sp_start_adv`

广播参数配置与启动逻辑：

```c
static void sp_start_adv(void)
{
    // 1. 动态构建扫描响应数据（设备名称 + 发射功率）
    uint8_t name_len = strlen(g_ble_dev_name);
    scan_rsp_data[0] = name_len + 1;
    scan_rsp_data[1] = GAP_ADVTYPE_LOCAL_NAME_COMPLETE;
    memcpy(&scan_rsp_data[2], g_ble_dev_name, name_len);
    
    // 2. 配置广播参数
    gap_adv_param_t adv_param = {
        .adv_mode = GAP_ADV_MODE_UNDIRECT,      // 可连接可扫描
        .adv_addr_type = GAP_ADDR_TYPE_PUBLIC,  // 公共地址
        .adv_chnl_map = GAP_ADV_CHAN_ALL,       // 37/38/39 全通道
        .adv_filt_policy = GAP_ADV_ALLOW_SCAN_ANY_CON_ANY,
        .adv_intv_min = 600,    // 375ms
        .adv_intv_max = 600,
    };
    gap_set_advertising_param(&adv_param);
    
    // 3. 设置广播数据与扫描响应数据
    gap_set_advertising_data(adv_data, sizeof(adv_data));
    gap_set_advertising_rsp_data(scan_rsp_data, scan_rsp_len);
    
    // 4. 启动广播（持续时间 0 = 无限）
    gap_start_advertising(0);
}
```

### 2.3 连接管理流程

#### 2.3.1 连接建立

```
手机发起连接请求
    ↓
BLE 协议栈处理物理层/链路层握手
    ↓
触发 GAP_EVT_SLAVE_CONNECT 事件
    ↓
app_gap_evt_cb() 回调处理
    ↓
启动连接参数更新定时器（4秒后请求更优参数）
```

**关键代码**（`ble_simple_peripheral.c`）：

```c
case GAP_EVT_SLAVE_CONNECT:
{
    co_printf("slave[%d],connect. link_num:%d\r\n", 
              p_event->param.slave_connect.conidx,
              gap_get_connect_num());
    // 4秒后请求更新连接参数：interval=15ms, latency=0, timeout=6s
    os_timer_start(&update_param_timer, 4000, 0);
}
break;
```

#### 2.3.2 连接参数更新

```c
static void ble_conn_param_timer_cb(void *arg)
{
    (void)arg;
    // 请求连接参数：min=12(15ms), max=12(15ms), latency=0, timeout=600(6s)
    gap_conn_param_update(0, 12, 12, 0, 600);
}
```

#### 2.3.3 连接断开

```c
case GAP_EVT_DISCONNECT:
{
    co_printf("Link[%d] disconnect,reason:0x%02X\r\n",
              p_event->param.disconnect.conidx,
              p_event->param.disconnect.reason);
    // 停止连接参数更新定时器
    os_timer_stop(&update_param_timer);

    gap_start_advertising(0);
}
break;
```

### 2.4 GATT 服务数据交互流程

#### 2.4.1 数据接收（手机→设备）

```
手机 Write 数据到 RX 特征值 (0xFF11)
    ↓
BLE 协议栈产生 GATTC_MSG_WRITE_REQ 事件
    ↓
ESAIR_gatt_msg_handler() 分发到 esair_gatt_write_cb()
    ↓
判断 att_idx：
    - ESAIR_ATT_IDX_CHAR_CFG_NOTI：更新 CCC 使能标志
    - ESAIR_ATT_IDX_CHAR_VALUE_RX：投递到应用任务
    ↓
app_task_send_event(APP_EVT_BLE_DATA_RECEIVED, data, len)
    ↓
应用任务处理 → protocol_parse_frame() → protocol_process_frame()
```

**关键代码**（`ESAIRble_service.c`）：

```c
static void esair_gatt_write_cb(uint8_t *write_buf, uint16_t len, 
                                uint16_t att_idx, uint8_t conn_idx)
{
    co_printf("ESAIR BLE RX: len=%d\r\n", len);
    
    if (att_idx == ESAIR_ATT_IDX_CHAR_CFG_NOTI) {
        // CCC 写入处理：使能/禁用 Notify
        if (len >= 2) {
            uint16_t ccc = (uint16_t)write_buf[0] | ((uint16_t)write_buf[1] << 8);
            ESAIR_link_ntf_enable = (ccc == 0x0001);
        }
        return;
    }
    
    // RX 数据投递到应用任务
    app_task_send_event(APP_EVT_BLE_DATA_RECEIVED, write_buf, len);
}
```

#### 2.4.2 数据发送（设备→手机）

```
应用层调用 protocol_send_buffer(..., PROTOCOL_SRC_BLE)
    ↓
ESAIR_gatt_report_notify(conidx, p_data, len)
    ↓
检查 ESAIR_link_ntf_enable 标志（必须客户端使能 CCC）
    ↓
gatt_notification(ntf) 调用 BLE 协议栈
    ↓
手机收到 Notify 数据
```

**关键代码**（`ESAIRble_service.c`）：

```c
void ESAIR_gatt_report_notify(uint8_t conidx, uint8_t *p_data, uint16_t len)
{
    if (ESAIR_link_ntf_enable)  // 必须客户端使能 CCC 后才能发送
    {
        gatt_ntf_t ntf = {
            .conidx = conidx,
            .svc_id = ESAIR_svc_id,
            .att_idx = ESAIR_ATT_IDX_CHAR_VALUE_TX,
            .data_len = len,
            .p_data = p_data
        };
        gatt_notification(ntf);
        co_printf("ESAIR BLE Notify sent, len=%d\r\n", len);
    }
}
```

### 2.5 动态更新设备名称流程

```
应用层调用 ble_update_device_name(new_name, name_len)
    ↓
验证名称合法性（长度、字符集）
    ↓
保存到 Flash（若 W25Q 存在）
    ↓
更新全局变量 g_ble_dev_name
    ↓
gap_stop_advertising() 停止当前广播
    ↓
gap_set_dev_name() 更新协议栈设备名称
    ↓
重建扫描响应数据（scan_rsp_data）
    ↓
gap_set_advertising_rsp_data() 更新广播数据
    ↓
gap_start_advertising(0) 重启广播
```

## 3. 关键参数与配置

### 3.1 枚举与结构体（`gap_api.h` / `gatt_api.h`）

- **`gap_event_type_t`**：GAP 事件类型枚举，包括连接、断开、参数更新、MTU 交换等。
- **`gap_adv_param_t`**：广播参数结构体，包含广播模式、地址类型、通道映射、间隔等。
- **`gap_security_param_t`**：安全参数结构体，包含 MITM、安全连接、IO 能力、配对模式等。
- **`gatt_attribute_t`**：GATT 属性定义结构体，包含 UUID、属性、最大长度、数据指针。
- **`gatt_msg_t`**：GATT 消息事件结构体，包含事件类型、连接索引、服务 ID、属性索引等。
- **`gatt_ntf_t`**：GATT 通知发送结构体，包含连接索引、服务 ID、属性索引、数据长度与指针。

### 3.2 本工程实际配置小结

| 项 | 取值 |
|----|------|
| 工作模式 | Peripheral（从机） |
| 广播模式 | `GAP_ADV_MODE_UNDIRECT`（可连接可扫描） |
| 广播间隔 | 600 × 0.625ms = **375ms** |
| 广播通道 | 37、38、39（全通道） |
| 地址类型 | `GAP_ADDR_TYPE_PUBLIC`（公共地址） |
| 发射功率 | **0dBm** |
| 设备外观 | `GAP_APPEARE_GENERIC_RC`（通用遥控器） |
| 安全模式 | Just Works（无 MITM、无输入输出） |
| 配对模式 | `GAP_PAIRING_MODE_WAIT_FOR_REQ`（等待请求） |
| 绑定存储 | Flash 地址 `0x7D000`，最多 8 个设备 |
| 连接参数 | Interval: 15ms, Latency: 0, Timeout: 6s |
| MTU 大小 | 默认 23 字节（可协商至更大） |
| 服务 UUID | `02F00000-0000-0000-0000-000000FE0010` |
| 单包最大 | **128 字节**（`ESAIR_MAX_DATA_SIZE`） |

## 5. 数据处理逻辑

### 5.1 协议帧格式（V2.2）

BLE 通道使用与 UART/MQTT 统一的二进制协议帧格式：

```
| 字段 | 长度 | 说明 |
|------|------|------|
| Frame Header | 2B | 0x5A 0x7A |
| Data Length | 1B | 5 + N（N 为数据长度） |
| Data Mark | 1B | 0x00=请求, 0x01=响应, 0x02=通知 |
| Command | 2B | 命令码（小端） |
| Dev Addr | 2B | 设备短地址（V2.2 新增） |
| Data | N B | 业务数据 |
| Checksum | 1B | 累加和校验 |
```

最大帧长度：**259 字节**（N ≤ 250）。

### 5.2 BLE 数据接收处理

```c
case APP_EVT_BLE_DATA_RECEIVED:
{
    uint8_t *data = (uint8_t *)param->param;
    uint16_t len = param->param_len;
    
    // 解析协议帧
    protocol_frame_t frame;
    if (protocol_parse_frame(data, len, &frame)) {
        // 地址过滤（V2.2）
        if (frame.data_mark == DATA_MARK_REQUEST && 
            !protocol_check_dev_addr(frame.dev_addr)) {
            protocol_send_response(frame->command, STATUS_ERROR_ADDR, PROTOCOL_SRC_BLE);
            break;
        }
        // 处理命令
        protocol_process_frame(&frame, PROTOCOL_SRC_BLE);
    }
}
break;
```

### 5.3 BLE 数据发送处理

```c
static void protocol_send_buffer(uint8_t *buffer, uint16_t len, uint8_t source)
{
    if (source == PROTOCOL_SRC_BLE) {
        // 通过 BLE Notify 发送
        ESAIR_gatt_report_notify(0, buffer, len);
    } else if (source == PROTOCOL_SRC_MQTT) {
        // MQTT 发送...
    } else if (source == PROTOCOL_SRC_UART) {
        // UART 发送...
    }
}
```

### 5.4 心跳包数据格式

心跳包（`CMD_HEARTBEAT`）通过 BLE 通道上报时，数据载荷（12 字节）：

```
| 字节 | 内容 |
|------|------|
| [0-3] | Power（float，IEEE754 小端，单位 W） |
| [4-7] | Temperature（float，IEEE754 小端，单位 °C） |
| [8] | Power status（0=OFF, 1=ON） |
| [9] | Mode（0=Auto, 1=Cold, 2=Hot, 3=Dry, 4=Fan, 5=Sleep） |
| [10] | Set temperature（16-30） |
| [11] | Wind speed（0-2） |
```

## 6. 错误处理机制

### 6.1 GAP 层错误处理

| 错误场景 | 处理方式 |
|----------|----------|
| 连接参数更新被拒绝 | 记录日志，维持当前参数 |
| 配对失败 | 记录日志，允许重新配对 |
| 绑定信息损坏 | `gap_bond_manager_delete_all()` 清除所有绑定 |
| 广播启动失败 | 记录日志，下次服务添加完成后重试 |

### 6.2 GATT 层错误处理

| 错误场景 | 处理方式 |
|----------|----------|
| CCC 未使能时发送 Notify | 静默丢弃（`ESAIR_link_ntf_enable` 检查） |
| 写入数据超长 | 协议栈自动拒绝，返回错误响应 |
| 服务未找到 | 返回 `GATT_SVC_ID_FAIL`（0xFF） |

### 6.3 协议层错误处理

| 错误码 | 值 | 触发条件 | 响应方式 |
|--------|-----|----------|----------|
| STATUS_SUCCESS | 0x00 | 命令执行成功 | 正常响应 |
| STATUS_ERROR_PARAM | 0x01 | 参数非法 | 返回错误码 |
| STATUS_ERROR_CMD | 0x02 | 命令不支持 | 返回错误码 |
| STATUS_ERROR_CHECKSUM | 0x03 | 校验和错误 | 丢弃帧，无响应 |
| STATUS_ERROR_BUSY | 0x04 | 设备忙（如红外发送中） | 返回错误码 |
| STATUS_ERROR_STORAGE | 0x05 | Flash 操作失败 | 返回错误码 |
| STATUS_ERROR_ADDR | 0x06 | 设备地址不匹配（V2.2） | 返回错误码 |
| STATUS_ERROR_FAIL | 0xFF | 通用失败 | 返回错误码 |

## 7. 使用注意事项

1. **CCC 使能要求**：设备向手机发送 Notify 数据前，必须等待手机写入 CCC（Client Characteristic Configuration）描述符使能通知；否则数据将被静默丢弃。

2. **连接参数影响**：默认广播间隔 375ms 平衡了功耗与连接速度；若需更快连接响应，可缩短广播间隔（最低 20ms），但会增加功耗。

3. **MTU 协商**：默认 ATT_MTU 为 23 字节（含 3 字节 ATT 头，有效载荷 20 字节）。若需传输更大数据包，应在连接后调用 `gatt_mtu_exchange_req` 协商更大 MTU（最大 247 字节）。

4. **设备名称长度**：设备名称最大长度 `BLE_NAME_MAX_LEN` 为 20 字节（不含 null 终止符）。更新名称时需确保不超过此限制。

5. **绑定设备数量**：绑定管理器最多支持 8 个配对设备。超过此数量时，新配对请求可能失败，需先删除旧绑定。

6. **协议帧地址过滤**：V2.2 协议新增设备短地址字段。收到请求帧时，若地址不匹配本地地址且非广播地址（0xFFFF），将返回 `STATUS_ERROR_ADDR` 错误。

7. **多通道并发**：BLE、MQTT、UART 三通道共享同一协议处理逻辑。同一命令从任一通道接收，响应将原路返回；心跳包则同时向所有已连接通道发送。

8. **OTA 升级注意**：OTA 升级期间，建议暂停其他大数据传输（如心跳包），避免影响升级数据流的实时性。
